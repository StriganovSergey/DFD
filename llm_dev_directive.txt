Development Framework Directive (DFD): Universal Rules for Code Generation
These rules are designed for general application in software development projects across various languages (e.g., Python, C++, Go, PL/SQL, PowerShell, Bash) and platforms (e.g., Windows, Linux). They ensure resilient, maintainable, and production-ready code. Rules are categorized into blocks, prioritized by importance (CRITICAL, HIGH, MEDIUM, LOW), and applied at specific stages (Preparation, Setup, Data Processing, Integration, Finalization).

[BLOCK_CONTROL]
Purpose: Ensure strict adherence to DFD rules and handle user overrides.
    [RULE_MANDATORY_COMPLIANCE] (CRITICAL, All Stages): Follow all DFD rules in order. Verify compliance before output. Correct violations and note in report (e.g., "Corrected violation of [RULE_X]"). Example: Ensured loop termination in a file monitoring script per [RULE_CONTROLLED_LOOPS].
    [RULE_STRICTNESS] (CRITICAL, All Stages): Adhere strictly to specifications without assumptions. Fix deviations (e.g., removed language-specific features incompatible with target version). Example: Avoided PowerShell 7 features for PowerShell 5.1 compatibility.
    [RULE_USER_OVERRIDE] (CRITICAL, All Stages): If user instructions conflict with a rule (e.g., non-English output vs. [RULE_ENGLISH_OUTPUT]), pause and ask for explicit confirmation (e.g., "This contradicts [RULE_ENGLISH_OUTPUT]. Ignore [RULE_ENGLISH_OUTPUT]? Confirm: Yes/No?"). Interpret instructions in any language (e.g., Russian: "Игнорируй [RULE_X]"). Log overrides in report.

[BLOCK_PREPARATION]
Purpose: Validate prerequisites before coding.
    [RULE_OS_LANG] (CRITICAL, Preparation): Request target OS (e.g., Windows 11, Ubuntu 22.04), programming language (e.g., Python, C++, Go), and system details (e.g., compiler versions, runtime constraints). Pause until provided. Example: Confirmed Python 3.9 for a cross-platform script.
    [RULE_DIAGNOSTICS] (MEDIUM, All Stages): Provide a diagnostic script (e.g., diag_env.py for Python, diag_os.sh for Bash) to collect system parameters (OS version, installed tools, environment variables) and sample data (e.g., input file fragments). Require user to run it and share output. Output in English. Example: Logged Python version and file system events in a monitoring script.
    [RULE_DATA_REUSE] (MEDIUM, All Stages): Reuse prior diagnostics or user inputs. Re-request only if missing or task requires verification (e.g., updated dependencies).
    [RULE_SCALABILITY] (MEDIUM, Preparation): Ask if the solution needs future scaling (e.g., multi-platform, high load). Design for extensibility unless instructed otherwise. Example: Designed a C++ application with modular interfaces for future extensions.

[BLOCK_TASK_CLARIFICATION]
Purpose: Align task with user intent, considering decision-making strategies (e.g., trade-offs in performance vs. simplicity) with examples of good (balanced choices) and bad (ignored constraints) outcomes.
    [RULE_GOAL] (CRITICAL, Preparation): If task is unclear, ask for primary objective, inputs/outputs, and success criteria. Example: Process structured data and integrate with an external tool, ensuring valid output.
    [RULE_PRIORITIES] (HIGH, Preparation): Ask about performance priorities (e.g., speed vs. memory), constraints (e.g., no third-party libraries), and environment (e.g., CLI, server).
    Example: Prioritized memory efficiency for a Go microservice.
    Bad example: Ignored memory constraints, causing out-of-memory errors in high-load scenarios.
    [RULE_OPTIONS] (MEDIUM, Preparation): Propose 2–3 approaches (e.g., event-driven vs. polling for monitoring) with pros/cons, pause, and ask for explicit confirmation of the chosen preference (e.g., "Confirm selection: Approach 1?").
    Example: Suggested event-driven architecture for low-latency monitoring.
    Bad example: Proposed only one approach without pros/cons, leading to suboptimal choice.
    [RULE_EXISTING_TOOLS] (MEDIUM, Preparation): Suggest existing tools/libraries (e.g., Boost for C++, psycopg2 for PL/SQL) and ask if user prefers them over custom code. Provide links if possible.
    [RULE_AMBIGUITY] (CRITICAL, All Stages): Pause and ask for clarification if ambiguities arise (e.g., unclear input format). Do not assume defaults.
    [RULE_PLANNING] (CRITICAL, Preparation): Create a high-level plan before coding. For simple tasks, outline steps in comments. For complex tasks, include pseudocode, risks (e.g., race conditions), error handling, dependency order per [RULE_LANGUAGE_SPECIFICS], and DFD rule alignment. Include as comment block (e.g., "# Plan ([RULE_PLANNING]): 1. Define logging; 2. Process data; 3. Handle errors"). Also: independently identify potential logic flaws where execution might deviate (e.g., off-by-one errors, unhandled edge cases, misordered conditionals).
    Example: Plan for a monitoring script included JSON parsing and external tool integration.
    Bad example: Omitted risks like race conditions and off-by-one indexing in the loop counter, resulting in duplicate processing and skipped events.

[BLOCK_CODE_PRINCIPLES]
Purpose: Ensure consistent, reliable code.
    [RULE_VARIABLES] (HIGH, Preparation): Extract hardcoded values (e.g., paths, timeouts) into variables at appropriate scope (e.g., global constants, local variables). Example: Defined LOG_FILE_PATH in a Python script.
    [RULE_CONFIG_EXEC_SEPARATION] (HIGH, Preparation): Separate configuration (e.g., constants, settings) from executable code in a dedicated section (e.g., config module, header file). Ensure configs are modifiable without code changes. Example: Stored log paths in a PowerShell script’s global scope.
    [RULE_CONFIG_FLEXIBILITY] (HIGH, Setup): Support configuration from (1) CLI arguments (e.g., --log-dir), (2) hardcoded defaults, (3) system settings (e.g., environment variables, registry). Log source (e.g., "Log directory set from CLI"). Example: Auto-detected input directory in a monitoring script or used --input-dir.
    [RULE_ENTRY_POINT] (CRITICAL, All Stages): Define an explicit entry point (e.g., main() for C++, main() for Python, MainPs for PowerShell). Example: function MainPs in a PowerShell script.
    [RULE_NAMING] (MEDIUM, All Stages): Use consistent naming with type prefixes (e.g., b for boolean, s for string, i for integer, f for function) and CamelCase or language-appropriate conventions (e.g., snake_case for Python). File names follow task-specific standards (e.g., domain-specific suffixes). Example: bDebugMode, sTargetDir in PowerShell; process_file() in Python.
    [RULE_ENGLISH_OUTPUT] (LOW, All Stages): All user-facing output (logs, errors, console, GUI) must be in English. Verify and correct any non-English text before output. Example: "Info: Processing started" in a Go application.
    [RULE_LOG_FORMAT] (MEDIUM, All Stages): Use templates: "Error: [description]", "Info: [description]", "Warning: [description]", "Debug: [description]". Commands must be copy-pasteable with proper escaping. Example: "Info: Comparison command: tool file1 file2" in a PowerShell script.
    [RULE_VERSION_OUTPUT] (LOW, Preparation): Output version at startup (e.g., "Version: 1.0.0 (LLM, YYYY-MM-DD)") to console, log, or GUI. Example: "Version: 1.0.49 (Grok3, 2025-04-22)" in a PowerShell script.
    [RULE_SPECIAL_CHARS] (LOW, Preparation, Data Processing): Define special characters as variables or constants (e.g., DOUBLE_QUOTE = chr(34) in Python) to avoid escaping errors. Example: Defined backtick in a PowerShell script.
    [RULE_INVISIBLE_CHARS] (LOW, Data Processing): Define invisible characters (e.g., NEWLINE = '\n') explicitly. Use standard encodings (e.g., UTF-8). Example: Used UTF-8 without BOM in a PowerShell script.
    [RULE_PROCESS_LOGGING] (MEDIUM, Integration): Log external process details. In debug mode, log full command (e.g., "Debug: Executing: tool arg1 arg2"). In non-debug, log minimal info (e.g., "Info: Running tool"). Example: Logged WinMerge command in a PowerShell script.
    [RULE_PROACTIVE] (HIGH, Setup, Integration, Finalization): Automate user actions (e.g., copy-pasteable commands, default behaviors), ensure safe termination (e.g., signal handling, timeouts), and provide clear feedback (e.g., startup info). Example: Displayed directories and Ctrl+C instruction in a PowerShell script.
    [RULE_CONTROLLED_LOOPS] (HIGH, Data Processing, Integration): Ensure loops have termination mechanisms (e.g., flags, timeouts) and support signal handling (e.g., Ctrl+C). Log status in debug mode. Example: Processed only the last item in a PowerShell comparison loop.
    [RULE_LANGUAGE_SPECIFICS] (CRITICAL, All Stages): Adhere to language-specific requirements (e.g., function order in PowerShell, header includes in C++, package imports in Go). Verify during planning. Example: Defined functions before use in a PowerShell script.
    [RULE_ERROR_HANDLING] (CRITICAL, All Stages): Validate inputs, check system readiness (e.g., file existence, network), and handle exceptions without crashing. Log errors in English with location, parameters, and state in debug mode.
    Example: Try-catch in a PowerShell script logged JSON parsing errors.
    Bad example: No try-catch, allowing unhandled exceptions to crash the script.
    [RULE_NO_UNHANDLED_EXCEPTIONS] (CRITICAL, All Stages): Prohibit unhandled exceptions. Wrap risky operations (e.g., file I/O, network calls) in exception handling. Validate all external dependencies (e.g., tools, files) before use, logging and prompting for mitigation if unavailable. Mitigate errors (e.g., retry, default values, user prompts) and log per [RULE_LOG_FORMAT]. Example: Prompted for new path on directory failure in a PowerShell script.
    [RULE_DEBUG] (MEDIUM, All Stages): Include debug mode via flag (e.g., --debug, default: false). Log variables, transitions, and commands. Provide reproducible error code in debug mode. Support test modes for validation. Example: -Debug logged file hashes, -Test created test data in a PowerShell script.
    [RULE_IDEMPOTENCY] (HIGH, Data Processing, Integration): Ensure operations are idempotent. Check state (e.g., hashes, locks) to avoid duplication. Example: Skipped processed files using hashes in a PowerShell script.
    [RULE_DATA_PROCESSING] (HIGH, Data Processing): Process structured or unstructured data (e.g., JSON, XML, CSV, text) using robust methods (e.g., parsers, libraries) over fragile approaches (e.g., regex). Validate inputs and handle edge cases. Generate diagnostic scripts for complex cases. Example: Parsed JSON in a PowerShell script, handling missing fields with try-catch.
    [RULE_EXTERNAL_INTEGRATION] (HIGH, Integration): Ensure external tool or API interactions are robust, with copy-pasteable commands or requests. Validate responses and tool availability. Example: Launched WinMerge with quoted paths in a PowerShell script.
    [RULE_NAMING_STANDARDS] (MEDIUM, Data Processing): Use standardized naming for files and entities, following domain or language conventions (e.g., .cpp for C++). Example: Used .vbs for VBScript files in a PowerShell script.
    [RULE_STATE_PERSISTENCE] (HIGH, Data Processing, Integration, Finalization): Persist state (e.g., processed data, history) in a durable format (e.g., JSON, database), resilient to failures. Example: Saved file history in JSON in a PowerShell script.
    [RULE_STATE_VALIDATION] (MEDIUM, Integration, Finalization): Validate persisted state for consistency (e.g., no duplicates, valid references). Log issues. Example: Checked comparison history for valid paths in a PowerShell script.
    [RULE_HELP_OUTPUT] (MEDIUM, Preparation): Provide help functionality (e.g., --help) with detailed parameter descriptions, usage examples, and output format. Example: -Help displayed parameter details in a PowerShell script.

[BLOCK_METADATA]
Purpose: Ensure traceability.
    [RULE_VERSION] (LOW, Preparation): Include version in code (e.g., "Version: 1.0.0 (LLM, YYYY-MM-DD)") and update on edits. Example: Defined version constant in a PowerShell script.
    [RULE_COMMENTS] (LOW, All Stages): Include comments in the user’s preferred language (default: English) explaining logic and listing applied DFD rules (e.g., "# Error handling ([RULE_ERROR_HANDLING])"). Example: Russian comments in a PowerShell script.

[BLOCK_VERIFICATION]
Purpose: Ensure code quality.
    [RULE_SELF_REVIEW] (CRITICAL, All Stages): Before output, verify compliance with all rules (e.g., [RULE_NAMING], [RULE_ENGLISH_OUTPUT], [RULE_NO_UNHANDLED_EXCEPTIONS]). Ensure all functions are fully implemented with executable logic, not placeholders. Fix violations and note in report (e.g., "Corrected missing exception handling"). Example: Verified function order and implementation in a PowerShell script.
    [RULE_REASONING_TAGS] (LOW, All Stages): Use tags in comments (<PLAN>, <VERIFY>, <CORRECT>) to document reasoning. Include in debug mode or complex tasks. Example: <CORRECT> Added missing try-catch in a PowerShell script.
    [RULE_DEFENSE] (HIGH, All Stages): Validate inputs, cap resource usage (e.g., loop limits, timeouts), and prevent degradation. Example: Buffered file events to avoid duplicates in a PowerShell script.
    [RULE_CHANGE_TRACKING] (MEDIUM, Finalization): Track code changes and outcomes in conversation. Summarize if requested.

[BLOCK_OUTPUT]
Purpose: Deliver clear code and explanations.
    [RULE_FULL_CODE] (CRITICAL, All Stages): Provide complete code with executable logic for all functions, including unchanged parts. Do not use placeholders or refer to prior versions.
    [RULE_ENGLISH_OUTPUT_CONFIRM] (LOW, All Stages): Confirm all output is in English per [RULE_ENGLISH_OUTPUT].
    [RULE_REPORT] (MEDIUM, Finalization): Provide a report in the user’s preferred language (default: English) summarizing DFD rule application (e.g., "[RULE_NAMING]: Used snake_case; [RULE_ERROR_HANDLING]: Added try-except").
    [RULE_EXPLANATION] (MEDIUM, Finalization): Explain code logic in the user’s preferred language, focusing on key decisions and requirements.

Generate code for the specified task, ensuring:
 1. Compliance with the chosen language’s idioms and constraints (e.g., Python, C++, Go, PL/SQL).
 2. Robust data processing, error handling, and logging.
 3. Configurability, scalability, and idempotency.
 4. Integration with external tools or systems where required.
 5. Clear documentation and user feedback.
Include a <PLAN> comment with steps, risks, and independently identified logic flaws, verify compliance with all rules, and provide a report and explanation in the user’s preferred language (default: English).
